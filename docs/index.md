### 项目简介

我选择了方向A：AI模型价格计算器。这个项目是一个前后端分离的AI API成本估算工具，用于比较海外主流AI API的文本模型和图像模型价格。用户可以输入每月请求次数、平均输入tokens、平均输出tokens、图片数量和图片尺寸，系统会根据不同模型的官方计费单位估算月度成本，并按成本从低到高排序。

项目前端使用Vue 3+ Vite实现，部署在Vercel；后端使用Node.js HTTP Server实现，部署在Render；价格缓存使用Upstash Redis。价格数据不是写死在前端，而是由后端抓取官方价格页面和搜索发现页，再调用DeepSeek API抽取结构化价格数据。为了避免频繁调用LLM，后端做了每日缓存：当天已有缓存时直接返回，手动刷新时才会重新请求。

### 技术栈

- Frontend：Vue 3、Vite、JavaScript、CSS
- Backend：Node.js HTTP Server
- Pricing Agent：DeepSeek Chat Completions API
- Cache：Redis（Upstash）
- Deploy：Vercel + Render

### 主要功能

- 文本模型成本估算：按输入tokens、输出tokens、月请求次数计算月成本。
- 缓存输入价格支持：如果模型有cached input price，可以选择使用缓存输入价估算。
- 图像模型成本估算：支持按图片计费和按megapixel计费。
- 价格来源页：单独展示模型价格、数据来源、查询日期和官网链接。
- 后端每日缓存：Redis优先，本地文件兜底。
- 移动端适配：页面在手机尺寸下不会出现明显布局溢出。

### 价格计算方式

文本模型通常以`USD / 1M tokens`计费，因此公式为：

```text
输入成本 = 输入 tokens * 月请求次数 / 1,000,000 * 输入单价
输出成本 = 输出 tokens * 月请求次数 / 1,000,000 * 输出单价
总成本 = 输入成本 + 输出成本
```

如果用户选择“使用缓存输入价格”，且该模型提供缓存输入价，则输入部分使用cached input price；否则继续使用普通输入价格。

图像模型根据计费单位分两种：

```text
按图片计费：总成本 = 图片数量 * 单图价格
按megapixel计费：总成本 = 图片数量 * 每张图 megapixel * megapixel 单价
```

### 已知限制

- 部分海外官网对服务器抓取有限制，后端可能无法稳定抓到所有官方页面。
- DeepSeek API额度不足时，价格刷新会失败，但已有缓存仍可继续使用。
- 免费Render实例会休眠，首次访问可能需要等待几十秒。
- 当前项目更偏成本预算，不包含模型质量、速度、上下文长度、成功率等综合评估。
- 价格数据依赖官方页面和LLM抽取结果，仍需要人工复核关键价格。

### 从0到上线：做一个AI API Cost Calculator的复盘

这次小项目我做的是一个AI API成本计算器，目标是让用户可以输入自己的使用量，然后估算不同AI模型API的月度成本。项目最终上线到了Vercel，后端部署在Render，价格缓存使用Upstash Redis。虽然功能看起来不复杂，但完整做下来之后，它覆盖了一个真实小产品从0到上线的大部分环节：需求拆解、数据来源、前后端分离、接口设计、缓存、环境变量、部署、跨域和复盘。

一开始我以为这个项目最难的是页面和计算公式。实际做起来后发现，公式确实是比较麻烦的一部分，如果把上下文长度、缓存命中率、批量请求折扣、免费额度和不同供应商计费边界等因素考虑进去，会是一个比较耗时的操作，但由于时间限制，我就选取了基础简单的实现方式。文本模型大多按每百万tokens计费，所以只要把输入tokens和输出tokens分开计算即可；图像模型则根据按图片或按megapixel计费分别处理。真正麻烦的是价格数据来源。最开始我把价格写死在前端，这样页面很快能跑起来，但它不符合“价格数据来自官方页面”的要求，也不利于后续更新。后来我把价格数据迁移到后端：前端只负责展示和计算，后端负责抓取官方页面，再调用DeepSeek抽取结构化数据。

这个过程中AI帮了我很多。它能快速生成Vue组件、Node接口、Redis缓存模块和部署说明，也能在我遇到Render、Vercel、Upstash配置问题时给出下一步操作。但AI也不是一直正确。比如它一开始倾向于使用固定模型白名单，这会导致新模型出现后不容易被捕获；后来我意识到应该固定“厂商和模型系列”，而不是固定具体版本号。再比如，后端一度设置成缺少OpenAI就直接报错，这在本地看起来是为了保证数据质量，但上线后反而让整个接口不可用。真实系统里，部分数据缺失时应该优先返回可用结果，同时把问题记录在metadata或日志中。

部署环节也让我踩了一些坑。Render后端需要配置DeepSeek API Key、Redis URL和CORS；Vercel前端只需要配置`VITE_API_BASE_URL`。这件事看起来简单，但如果把后端密钥放到前端环境变量里，就会有泄露风险。另一个坑是Vercel识别项目结构时，把仓库看成多服务项目。最后通过明确build command和output directory，让它只部署`frontend/dist`。Render免费实例会休眠，所以第一次打开页面比较慢，这也是免费部署平台的正常限制。

这次项目最大的收获是：AI可以显著提高编码速度，但不能代替工程判断。一个能上线的小工具，不只是页面能显示，还要考虑数据是否可信、密钥是否安全、缓存是否合理、部署是否稳定、失败时用户看到什么。下次如果再做类似任务，我会更早确定架构边界：前端只放公开逻辑，后端处理密钥和数据刷新，缓存先设计好，部署配置尽量写进文档或配置文件里。这样可以减少后期反复修改的成本。

## 第4部分：自评

1. 这个项目我最满意的一点：第一次完整一个项目从0到1的开发部署上线，以前都只是开发，没有部署过。
2. 最心虚的一点：价格数据依赖LLM从网页中抽取，虽然能跑，但长期稳定性和准确性还需要更严格的校验，而且目前功能过于简单，后续还需要优化。
3. 再给我8小时，我会优先修改当前的计算方式：除了按tokens和图片数量估算基础成本，还会加入上下文长度、缓存命中率、批量请求折扣、免费额度和不同供应商计费边界等因素；同时继续优化页面，比如增加更清晰的筛选、模型分组、错误状态说明和移动端细节，最后再补充价格刷新日志与人工复核入口。
4. 整个过程中我卡得最久的地方是“公式计算以及价格数据如何既自动更新又可信”，包括官方页面抓取失败、LLM抽取结果不稳定、缓存策略和部署环境差异，前后大约花了2小时左右。
5. 如果入职第一周做类似任务，我希望团队提供目标用户场景、必须支持的数据源清单、部署环境约束，以及对数据准确性的验收标准。
